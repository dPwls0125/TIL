API 사양 테스트에서 중요한 지점은 다음과 같습니다.

   1. 계약 준수 (Contract Adherence): API 사양(문서)에 정의된 대로 요청 및 응답 형식이 정확히 일치하는지 확인하는 것이 가장
      중요합니다. 이는 다음과 같은 세부 사항을 포함합니다.
       * HTTP 메서드: 올바른 HTTP 메서드(GET, POST, PUT, DELETE 등)가 사용되는지.
       * URL 경로: 정의된 경로(Path)가 정확한지.
       * HTTP 상태 코드: 성공 및 실패 시 예상되는 HTTP 상태 코드(예: 200 OK, 201 Created, 400 Bad Request, 404 Not Found, 500
         Internal Server Error)가 반환되는지.
       * 요청/응답 헤더: 필요한 헤더(예: Content-Type, Authorization)가 올바른지.
       * 요청/응답 바디 스키마: JSON, XML 등 바디의 구조, 필드 이름, 데이터 타입이 사양과 일치하는지. 필수 필드와 선택 필드가
         제대로 처리되는지.

   2. 데이터 유효성 검사 (Data Validation):
       * 입력 유효성: API가 유효하지 않거나 누락된 입력 데이터에 대해 올바르게 응답하는지 (예: 400 Bad Request와 함께 의미
         있는 오류 메시지 반환).
       * 출력 유효성: API가 항상 유효하고 예상된 형식의 데이터를 반환하는지.

   3. 예외 처리 (Error Handling):
       * 예상치 못한 상황(예: 데이터베이스 오류, 외부 서비스 장애)에서 API가 적절한 HTTP 상태 코드와 명확한 오류 메시지를
         반환하여 클라이언트가 문제를 이해하고 처리할 수 있도록 하는지.

   4. 보안 (Security):
       * 인증 및 권한 부여 메커니즘(예: 토큰, 세션)이 올바르게 작동하여 승인되지 않은 접근을 차단하는지.
       * 민감한 정보가 응답에 노출되지 않는지.

   5. 성능 (Performance) 및 확장성 (Scalability):
       * 반환되는 데이터 양이 합리적인지 (너무 많거나 적지는 않은지).
       * (초기 단계에서는 아니지만, 중요해지면) 특정 부하에서 API가 허용 가능한 응답 시간을 유지하는지.

   6. 사용성 (Usability):
       * API가 일관성 있는 디자인 패턴과 명명 규칙을 따르는지.
       * API 문서와 실제 구현이 일치하여 개발자가 쉽게 사용할 수 있는지.

  특히 지금의 맥락처럼 더미 데이터를 반환하는 컨트롤러의 경우, 계약 준수가 가장 중요합니다. 즉, 실제 비즈니스 로직이 구현되기
  전에 API의 외부 계약(인터페이스)이 정확하게 정의되고 준수되는지를 확인하는 데 중점을 두어야 합니다. 이를 통해
  프론트엔드/클라이언트 개발과 백엔드 개발이 병렬적으로 진행될 수 있으며, 통합 시 발생할 수 있는 인터페이스 불일치 문제를
  사전에 방지할 수 있습니다. 배열 요소의 구조와 유형을 테스트하는 것은 복잡한 데이터 구조에 대한 계약 준수를 보장하는 데
  직접적으로 기여합니다.
