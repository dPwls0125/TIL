#  I/O 처리방식 _ 인터럽트 구동 I/O VS DMA 

**운영체제 코드의 상당 부분**은 시스템의 안정성과 성능에 대한 중요성과 장치의 다양한 특성으로 인해 **I/O에 할애**된다.

```OS/Computer-System Organization.md``` 에 적어둔 인터럽트 구동 I/O 방식의 경우에는 대량의 데이터 이동에 사용될 때 높은 오버헤드가 발생할 수 있다.  

> **바이트(Byte)** 는 데이터의 최소 처리 단위(8bit)이고, **블록(Block)** 은 여러 바이트를 묶은 데이터 전송 단위(ex. 512B, 4KB)이다.

**예를 들어 1KB(1024바이트) 데이터를 디스크에서 메모리로 옮긴다고 해보자.**

### 인터럽트 구동방식 
```aiignore
1) 인터럽트 구동 방식에서는 1바이트 전송될 때마다 CPU에 인터럽트가 걸린다. (OS/Computer-System Organization.md 참조.)
2) CPU가 직접 그 1바이트를 받아서 메모리에 써야 해. 그러니까 1024번 인터럽트가 발생하고,     
3) 매번 CPU가 하던 일을 멈추고 → 컨텍스트 저장 → 데이터 옮기기 → 복귀하는 과정을 반복.
```

### DMA 방식 
```aiignore
1) CPU가 DMA 컨트롤러한테 "소스 주소, 목적지 주소, 크기 1024바이트" 이 정보만 알려주고 빠진다.  
2) DMA 컨트롤러가 CPU 없이 직접 버스를 통해 1024바이트를 메모리로 전송한다.  
3) 1024바이트 전송이 다 끝났을 때 딱 1번 인터럽트를 걸어서 "다 했어"라고 CPU에 알려준다.
```
**결과적으로**  
대량의 데이터를 전송해야하는 상황에서, 만약 장치 컨트롤러가 매번 CPU에 인터럽트를 발생시킨다면 CPU는 context switching이 대량 발생하며 오버헤드가 발생한다.  

반면, DMA 방식에서는 DMA 컨트롤러가 장치 컨트롤러의 버퍼와 메모리 사이에서 직접 데이터를 전송하고, 블록 전송이 완료된 시점에 딱 한번만 인터럽트가 발생하기 때문에 CPU의 오버헤드가 크게 줄어든다.  


**버스의 병목과 스위치 구조**  

DMA 전송 시 DMA 컨트롤러가 버스를 점유하면 CPU는 버스 접근이 불가하다(cycle stealing).
장치가 많아질수록 하나의 공유 버스에서 병목이 발생하기 때문에, 몇몇 고가의 시스템에서는 버스 대신 스위치 구조를 사용하여 여러 장치가 동시에 서로 다른 경로로 통신할 수 있게 한다. (ex. PCIe의 point-to-point 연결)


**3가지 I/O 처리 방식 정리**

- Polling (폴링) — CPU가 장치 상태를 계속 확인하면서 기다리는 방식 (busy waiting)
- Interrupt-Driven I/O (인터럽트 구동 I/O) — 장치가 준비되면 인터럽트로 CPU에 알려주는 방식
- DMA — DMA 컨트롤러가 데이터 전송을 대신하고, 완료 시 인터럽트로 알려주는 방식
